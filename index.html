01 — Fundamentals
What is a Deadlock?
A deadlock is a system state where a set of processes are permanently blocked — each process holds at least one resource and waits for another resource held by another process in the same set. No process can proceed, and the system stalls indefinitely without external intervention.
System Model
The system contains m resource types R₁…Rₘ, each with Wᵢ instances. Process lifecycle: Request → Use → Release. A deadlock forms when the Resource Allocation Graph (RAG) contains a cycle (single-instance) or when the Coffman matrix algorithm returns unfinished processes.
Safe vs Unsafe State
A state is SAFE if there exists a safe sequence <P₁…Pₙ> where each Pᵢ can get all needed resources from: currently available + resources held by all Pⱼ (j < i). An unsafe state may lead to deadlock but is not necessarily deadlocked yet.
Three Handling Approaches
Prevention — eliminate one Coffman condition at design time.
Avoidance — Banker's Algorithm keeps system in safe states.
Detection & Recovery — allow deadlocks, detect via RAG/matrix, then recover.
Most real OSes use a combination of all three.
02 — Coffman's 4 Necessary Conditions
01
Mutual Exclusion
At least one resource must be held in a non-shareable mode. Only one process at a time can use the resource; others must wait. Inherent for devices like printers, mutexes, semaphores.
Cannot always be prevented
02
Hold and Wait
A process holding ≥1 resource is simultaneously waiting to acquire additional resources currently held by other processes. Prevention: require all resources upfront.
Can be prevented — low utilization cost
03
No Preemption
Resources cannot be forcibly taken from a process. They are released only voluntarily after the process has finished using them. Prevention: allow OS to preempt (e.g. CPU scheduling).
Can be prevented for some resources
04
Circular Wait
A closed chain P₁→P₂→…→Pₙ→P₁ where each Pᵢ holds a resource needed by P(i+1). Prevention: impose a total ordering on all resource types and require requests in order.
Most practical prevention target

Detection Algorithms
wait_for_graph.py O(V + E) 
# DFS Cycle Detection on Wait-For Graph
# Single-instance resources only
# An edge Pi→Pj means Pi waits for Pj

def detect_deadlock_wfg(processes, wait_for):
    WHITE, GRAY, BLACK = 0, 1, 2
    color      = {p: WHITE for p in processes}
    deadlocked = []

    def dfs(u, path):
        color[u] = GRAY
        path.append(u)

        for v in wait_for.get(u, []):
            if color[v] == GRAY:        # back-edge
                idx = path.index(v)
                deadlocked.extend(path[idx:])
                return True
            if color[v] == WHITE:
                if dfs(v, path): return True

        path.pop()
        color[u] = BLACK
        return False

    for p in processes:
        if color[p] == WHITE:
            dfs(p, [])

    return list(set(deadlocked))
bankers_safety.py O(n²m) 
# Banker's Safety Algorithm
# Returns safe sequence or None

def bankers_safety(need, alloc, avail):
    n, m   = len(need), len(avail)
    work   = avail[:]
    finish = [False] * n
    seq    = []

    while len(seq) < n:
        found = False
        for i in range(n):
            if not finish[i]:
                can = all(need[i][j] <= work[j]
                          for j in range(m))
                if can:
                    for j in range(m):
                        work[j] += alloc[i][j]
                    finish[i] = True
                    seq.append(f"P{i}")
                    found = True
        if not found:
            return None  # UNSAFE

    return seq  # Safe sequence
multi_instance_detection.py O(n²m) 
# Multi-Instance Deadlock Detection
# Coffman matrix-based algorithm
# Handles multiple resource instances

def detect_multi(alloc, request, avail):
    n = len(alloc)
    m = len(avail)
    work   = avail[:]
    finish = [False] * n

    # Procs with no outstanding requests
    for i in range(n):
        if all(request[i][j] == 0
               for j in range(m)):
            finish[i] = True

    changed = True
    while changed:
        changed = False
        for i in range(n):
            if not finish[i]:
                can = all(
                    request[i][j] <= work[j]
                    for j in range(m)
                )
                if can:
                    for j in range(m):
                        work[j] += alloc[i][j]
                    finish[i] = True
                    changed   = True

    dead = [i for i in range(n)
            if not finish[i]]
    return dead  # deadlocked indices

When to Trigger Detection?
▸ Periodically — every fixed interval (e.g. every 5 minutes)
▸ On block — whenever a process blocks on a resource request
▸ CPU heuristic — when CPU utilization drops near 0%
▸ On demand — manual operator trigger

Trade-off: frequent detection → low deadlock latency, high CPU cost.
Infrequent detection → less overhead, longer deadlock duration.
Recovery Strategies
☠️
Terminate All
Abort all deadlocked processes simultaneously. Simple but highest data loss.
High cost · Immediate
🎯
Terminate One-by-One
Kill cheapest victim, re-detect, repeat until deadlock is resolved.
Medium cost · Iterative
⚡
Resource Preemption
Forcibly take resources from a low-priority victim. Victim rolls back to waiting.
Medium cost · Selective
↩️
Checkpoint Rollback
Roll process back to last saved checkpoint. Requires checkpointing overhead.
Low data loss · High overhead
⏳
Wait-Die
Older process waits; younger process is aborted. Timestamp-based, no starvation.
No starvation · DB-friendly
🗡️
Wound-Wait
Older wounds (preempts) younger; younger waits for older. Preemptive variant.
Preemptive · Low latency
🏆
Priority Victim
Cost function considers priority, age, resources held. Kills minimum-cost victim.
Optimal selection
⚙️
Combined Approach
Partition resources — prevention for some, avoidance for others, detection for rest.
Production-grade
